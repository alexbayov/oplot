## Lessons learned (применять на M7)

Из M3+M4+M5+M6 в порядке свежести:

- **QA fix cherry-pick pattern (M6)**: если QA Acceptance находит runtime bug и патчит его в test-octopus branch, PM cherry-pick'ает fix в Engineer's role-PR (`git cherry-pick` локально → REST API push), а не self-merge role-PR с QA branch. Audit trail чище.
- **QA Acceptance PR merge order (M6)**: PM мерджит QA Acceptance PR **последним** (после role-PR'ов) — net delta сводится к QA report file. Этот паттерн чище для audit trail.
- **4-параллельный launch (M6)**: Content + Engineer + Artist + QA Acceptance preview могут запуститься одновременно после QA Spec APPROVE; QA Acceptance дожидается role-PR Ready перед octopus-merge dry-run.
- **DoD-precision (M3 → M4 → M5 → M6)**: exact numbers (9 zones, 80 items, 43 recipes, 12 SFX, 18 anim, 194 vitest), не «≥» / «около». Cross-spec misalign = blocker, не silent-fix.
- **Status-sync PR pattern (M6)**: PM может опубликовать промежуточный `pm/m7-status-sync` PR для синхронизации dashboards под факт GitHub без блокировки production. Self-merge по делегации Alex'а.
- **Octopus-merge dry-run в QA Acceptance (M3 → M6)**: QA локально merge'ит все 3 role-PR (`qa/m7-acceptance-test`) и прогоняет 3 Gate'а перед PR submission.
- **Token-budget (M2)**: план role-сессии **строго 5-7 пунктов**. Все role-сессии M3-M6 уложились — работает.
- **Recovery-safe (M2 → M6)**: ранний Draft PR + commit/push после каждого подшага + PR Recovery block в description. Все role-сессии M3-M6 следовали — recovery срабатывает в 100% случаев.
- **PAT-hygiene (M2 → M6)**: PAT только в Authorization header / GIT_ASKPASS env var, никогда в URL / echo / print / commit body. PM-сессии используют `$GITHUB_PAT_ALEXBAYOV` или `$git_pat` через `/tmp/git-askpass.sh`.
- **Git-proxy 403 workaround (M4 → M6)**: между PM-сессиями смена PAT может вызвать 403 от `git-manager.devin.ai/proxy/github.com/`. Fix: `git -c url.https://github.com/.pushInsteadOf=https://github.com/ -c credential.helper= push` с GIT_ASKPASS, отдающим $GITHUB_PAT_ALEXBAYOV в password-prompt.
- **Anti-scope discipline (M3 → M6)**: каждый kickoff явно перечисляет, что НЕ входит в веху. M7 anti-scope (см. секция выше) фиксирует Yandex SDK, новые mobs/bosses, T5, modular, skill tree, faction-rep, real-time timers, music, voice, redesign.
- **Cross-spec расхождение (M3 → M6)**: GDD vs balance vs content vs assets вычислены по разному = blocker; эскалация в PM, НЕ резолвить самовольно в role-сессии. На M7 особенно критично из-за объёма (45 items + 25 recipes + 18 anim + 12 SFX).

## Особые риски M7 (Variant B большая веха)

1. **Контентный объём** — Content M7 единая роль готовит +6 zones (18 depth configs) + +45 items + +25 recipes + +12 SFX registry = крупнейшая Content role-сессия за всё время проекта. Mitigation: token-budget план 5-7 пунктов с явной разбивкой по типам контента; ранний Draft PR; commit/push после каждой группы (zones → items → recipes → sfx).
2. **Audio первый раз в проекте** — Engineer M7 setup'ит Phaser audio system впервые. Mitigation: tested API surface (`playSFX`, `setMasterVolume`, `setMuted`), persistence в GameState, Settings UI minimal, audio assets (OGG/MP3) предоставляет Artist.
3. **Balance tuning без regressions** — изменение M2-M6 baseline numbers рискует сломать существующий M2-M6 flow. Mitigation: balance §M7 tuning table fixates before/after с rationale; QA Acceptance regression smoke с manual checklist; vitest balance constants validation.
4. **Asset budget** — M7-add ≤ 320 KB (6 zone bg + 6 zone icons + 30 item icons + 12 SFX) → project total ≤ 900 KB. Это +250 KB к M6 budget. Mitigation: zone bg использует Pillow с low-detail (mid-tier ~30 KB each), item icons compact (~2 KB each), SFX короткие OGG/MP3 (≤ 8 KB each, total ≤ 96 KB).
5. **6 параллельных контент-источников** — zones / items / recipes / SFX / mob-AI mapping / unlock conditions могут разъехаться. Mitigation: GD M7 amendment даёт single source of truth таблицу (zone → mob → resource → item → recipe map), QA Spec проверяет cross-ref до production start.
