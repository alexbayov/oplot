# Партитура: как команда работает по вехе

Документ описывает **полный цикл одной М-вехи** + **шаблоны промтов** на каждую роль. Это «дирижёрская партитура» — кому что PM даёт, когда, и что ждёт в ответ.

Связанные документы:
- `staff/TEAM.md` — кто за что отвечает
- `staff/PLAN.md` — какие вехи и в каком порядке
- `staff/ORCHESTRATION.md` — playbook для PM
- `staff/roles/*.md` — детализация по ролям

---

## 1. Два уровня цикла

Работа идёт на **двух уровнях**:

- **Внешний цикл (М-веха)** — порядок ролей: GD → QA-spec → Eng+Content+Artist → QA-accept → закрытие. Описан в §1.1.
- **Внутренний цикл (одна Devin-сессия)** — что делает каждая сессия с момента старта до PR. Описан в §1.2.

Каждый шаг внешнего цикла — это **отдельная Devin-сессия**, которая внутри проходит 6 шагов внутреннего цикла.

---

## 1.1. Внешний цикл одной М-вехи (7 шагов)

```
┌────────────────────────────────────────────────────────────┐
│  ШАГ 0: PM готовит веху                                     │
│  - Пишет staff/prompts/M{N}.md (ТЗ по ролям)               │
│  - Пишет kickoff/M{N}-*.md + handoff/M{N}-*.md (6 пар)     │
│  - Коммитит в main                                          │
│  - Говорит Alex'у: «M{N} готова к запуску»                  │
└────────────────────────────────────────────────────────────┘
                        │
                        ▼
┌────────────────────────────────────────────────────────────┐
│  ШАГ 1: Game Designer (последовательно)                     │
│  Alex копипастит kickoff/M{N}-GD.md → новая Devin-сессия   │
│  GD: читает → план → апрув PM → пишет GDD.md → PR          │
│  ⏱ ~1-3 часа                                                │
└────────────────────────────────────────────────────────────┘
                        │
                        ▼
┌────────────────────────────────────────────────────────────┐
│  ШАГ 2: QA spec review (последовательно)                    │
│  Alex копипастит kickoff/M{N}-QA-SPEC.md → новая сессия    │
│  QA: проверяет GDD → APPROVE / CHANGES_REQUESTED            │
│  ⏱ ~30-60 мин                                               │
└────────────────────────────────────────────────────────────┘
                        │ (только после APPROVE)
                        ▼
┌────────────────────────────────────────────────────────────┐
│  ШАГ 3: Eng + Content + Artist (ПАРАЛЛЕЛЬНО, 3 сессии)     │
│  Alex копипастит 3 kickoff'а → 3 новые сессии               │
│  Каждый: план → апрув → работа → PR                         │
│  Не пересекаются по файлам:                                 │
│    Engineer → src/, package.json                             │
│    Content  → content/*.json                                 │
│    Artist   → assets/, docs/style-guide.md                   │
│  ⏱ ~1-2 дня                                                 │
└────────────────────────────────────────────────────────────┘
                        │
                        ▼
┌────────────────────────────────────────────────────────────┐
│  ШАГ 4: QA acceptance (последовательно)                     │
│  Alex копипастит kickoff/M{N}-QA-ACCEPT.md → новая сессия  │
│  QA: тестирует все 3 PR → APPROVE / CHANGES                 │
│  ⏱ ~1-2 часа                                                │

                        │ (после APPROVE)

┌────────────────────────────────────────────────────────────┐

│  - Готовит промты M{N+1}                                    │
│  - Говорит Alex'у: «M{N} закрыта, готов к M{N+1}?»         │
└────────────────────────────────────────────────────────────┘
```

## 1.2. Внутренний цикл одной Devin-сессии (6 шагов)

```
1. ОЗНАКОМЛЕНИЕ → клонирует репо, читает handoff + status + PROCESS + PLAN + свою роль
2. ПЛАН          → 5-10 пунктов, блокирующее сообщение Alex'у
                   «План готов, жду апрува PM»
3. ЖДИ АПРУВА    → Alex форвардит план PM, PM возвращает через Alex'а либо
                   «апрув плана», либо список правок. До апрува сессия НЕ работает.
4. ВЫПОЛНЕНИЕ    → строго по утверждённому плану + по handoff'у.
                   Если обнаружена дыра — стоп, вопрос Alex'у.
5. PR            → ветка m{N}/<role>-<scope>, описание + чек-лист + recovery block,
                   БЕЗ self-merge. Сессия сообщает Alex'у ссылку на PR.
6. ОТЧЁТ         → обновляет только staff/status/{ROLE}.md, блокирующее Alex'у:
                   «PR <ссылка>, готов к ревью PM».
```

**Никакой self-merge.** Любая роль создаёт PR и НЕ мерджит. Мерджит только PM.

**Self-update правило:** роль обновляет только свой status-файл и PR recovery block. Общий dashboard `staff/status/M{N}.md`, `PLAN.md`, `CHANGELOG.md` и чужие `status/*.md` обновляет только PM.

## 2. Правила оркестрации

### 2.1. Что строго последовательно
- **GD-спека → QA-ревью спеки → разработка.** Никогда не пишем код без апрува спеки.
- **Разработка → QA-приёмка.** Никогда не считаем веху «готовой» без чек-листа QA.
- **QA-приёмка → закрытие.** Веха закрывается только когда чек-лист пройден.

### 2.2. Что параллельно
- **Engineer + Content Designer + Artist** работают параллельно (шаг 3), потому что их артефакты независимы (src/ vs content/ vs assets/).
- **QA пишет чек-листы будущей вехи M{N+1}**, пока команда делает M{N}.

### 2.3. Кто кого может блокировать
- **Game Designer блокирует Engineer / Content / Artist**, если нет утверждённой спеки.
- **QA блокирует всех** — PR не мерджится, пока чек-лист не пройден.
- **Заказчик блокирует PM** на закрытии вехи и при попытке выйти за anti-scope.

### 2.4. Эскалация к Заказчику

PM эскалирует к Заказчику (dev07) в этих случаях:
1. Спека GD конфликтует с PLAN.md.
2. QA вечно не апрувит — третий круг правок.
3. Технический блокер: нужна сторонняя зависимость / SDK-ключ.
4. Сроки уходят больше чем на +30% от плана.
5. Кто-то хочет нарушить anti-scope.

## 3. Recovery-протокол (обрыв сессии)

Если Devin-сессия оборвалась (токены, ошибка, таймаут):

1. PM проверяет `staff/status/M{N}.md` — текущий gate и blockers.
2. PM проверяет `staff/status/{ROLE}.md` — что успело обновиться.
3. PM проверяет PR recovery block, если PR уже есть.
4. PM создаёт **recovery kickoff**: «Ты продолжаешь работу. Прочитай status/{ROLE}.md и recovery block. Доделай то, что в "Что НЕ сделано"».
5. Alex копипастит recovery kickoff → новая сессия подхватывает.

## 4. Шаблоны промтов

Все промты имеют одинаковую структуру:

```
ROLE: {роль}
MILESTONE: M{N} — {название}
REPO: https://github.com/alexbayov/oplot
HANDOFF: staff/handoff/M{N}-{ROLE}.md

PRECONDITIONS:
- {что должно быть готово ДО запуска}

DELIVERABLES:
- {файл/артефакт 1}
- {файл/артефакт 2}

FORBIDDEN:
- {чего НЕ делать}

STEPS:
1. Клонируй репо. Прочитай handoff + status/{ROLE}.md + PROCESS.md + PLAN.md + свою роль.
2. Напиши план (5-10 пунктов), отправь Alex'у блокирующим.
3. После апрува — выполняй строго по плану.
4. Создай ветку m{N}/<role>-<scope>, оформи PR с recovery block. Сам НЕ мерджи.
5. Обнови только staff/status/{ROLE}.md.
6. Сообщи Alex'у: «PR <ссылка>, готов к ревью PM».
```

Конкретные промты для каждой вехи — в `staff/prompts/M{N}.md`.
Kickoff-версии (для копипаста) — в `staff/kickoff/M{N}-{ROLE}.md`.
Полные брифинги — в `staff/handoff/M{N}-{ROLE}.md`.
