# Ledger — durable per-PR state для Codex

> **Назначение.** Снять «GO в Slack после каждого мержа» с критического пути.
> Codex читает `CURRENT.md` в начале каждой итерации и сам понимает: «что я делаю
> следующим, на каком main я стою, какие D-форки пакета уже резолвлены, какие
> guard'ы должны зелёнеть». QA (Ross) обновляет `CURRENT.md` как часть мерж-ритуала.
>
> Slack-тред остаётся для исключений: архитектурные форки в середине пакета,
> блокеры, человеческий call. Рутинный «след. PR → preflight → PR → QA → merge»
> едет по ledger без Slack-handshake'а.

---

## Зачем

До ledger'а контракт был:

```
preflight → GO в Slack → PR → QA → merge --no-ff → STOP, ждать следующего GO
```

Каждый STOP — это раунд-trip через Slack, ~часы паузы между PR'ами одного пакета.
В пакете на 5-6 PR это 5-6 пауз там, где архитектура уже резолвлена в preflight
пакета и нечего обсуждать.

После ledger'а:

```
Codex: читает staff/ledger/CURRENT.md
        → находит секцию "Следующий PR" с GO
        → пишет preflight (если ещё не написан) → PR → ждёт QA
QA:   мержит --no-ff
        → в том же ритуале обновляет CURRENT.md: помечает PR закрытым,
          переключает "Следующий PR" на следующий слот, либо ставит "пауза, жду".
        → коммитит обновление ledger'а (отдельный коммит или фолд-ин в merge commit).
Codex: следующий спавн читает CURRENT.md, видит новый GO, едет.
```

Slack нужен только когда:
- Пакет исчерпан → QA пишет в ledger «пакет закрыт, жду нового preflight» → пинг.
- Codex упёрся в архитектурный форк, которого не было в package-preflight → пинг.
- QA нашёл блокер, который меняет следующий PR → правка ledger'а + пинг.

## Контракт

### Codex (eng) — в начале каждой сессии

1. **Прочитать `staff/ledger/CURRENT.md` полностью.** Это первый файл, который ты
   читаешь после клона/checkout main. До этого никакой код не трогается.
2. Сверить `main HEAD` из `CURRENT.md` с реальным `git rev-parse main` локально.
   Если разъехались — `git fetch && git checkout main && git pull --ff-only`,
   потом перечитать ledger.
3. Найти секцию **«Следующий PR»**:
   - Если `Статус: GO` — едешь: preflight (если ещё нет) → ветка → код → PR.
   - Если `Статус: ОЖИДАНИЕ` — стоп. Не пиши код, не открывай PR. Можно
     прокомментировать в Slack «жду ledger», но обычно просто молчи.
   - Если `Статус: BLOCKED` — читай раздел «Блокер», отвечай в Slack по сути.
4. **Не редактируй `CURRENT.md` сам.** Это QA-only файл. Если хочешь что-то
   добавить (например, после твоего preflight'а — pre-resolved D-форк), делай
   это в `staff/implementation/<PR>-PREFLIGHT.md` и сошлись на него из PR'а.
   QA подтянет ссылку в ledger при мерже.

### QA (Ross) — после каждого merge

1. `git merge --no-ff` как обычно.
2. **В том же ритуале** обновить `CURRENT.md`:
   - Сдвинуть только что закрытый PR в раздел «Закрыто в этом пакете» с SHA
     merge commit'а и одной строкой «что сделано» (для будущей памяти).
   - Сдвинуть «Следующий PR» на следующий слот из package backlog. Заполнить
     поля: slug, цель в одной строке, ссылка на preflight-раздел в
     package preflight'е, baseline gates (если бамп ожидается — записать
     ожидание), статус `GO`.
   - Обновить `main HEAD` сверху файла.
3. Закоммитить ledger в main отдельным коммитом с сообщением вида
   `ledger: M18-PR1 closed, M18-PR2 next`.
4. **Если пакет исчерпан** — поставить «Следующий PR» в `Статус: ОЖИДАНИЕ`,
   пинг в Slack «пакет M18 закрыт, нужен preflight следующего».
5. **Если найден блокер, меняющий план** — поставить `Статус: BLOCKED`, заполнить
   раздел «Блокер» (что, почему, что нужно от человека). Пинг в Slack.

### Кто что владеет

| Файл | Владелец | Кто читает |
|---|---|---|
| `staff/ledger/CURRENT.md` | QA (Ross) — write | Все. Codex — на каждом спавне. |
| `staff/ledger/README.md` (этот файл) | QA — write | Codex — один раз для онбординга. |
| `staff/implementation/<PR>-PREFLIGHT.md` | Codex — write | QA — на review preflight'а. |
| `staff/implementation/PACKAGE-<id>-PREFLIGHT.md` | Codex — write, QA — review | Codex — между PR'ами пакета. |

## Что НЕ кладём в ledger

- **Дизайн-решения.** Они в `docs/redesign/`, package preflight'е, или
  per-PR preflight'е. Ledger ссылается, не дублирует.
- **Историю.** Полная история PR'ов в `staff/handoff/M*-*.md` и git log.
  Ledger держит только текущий пакет + одну строку про каждый закрытый PR пакета
  для контекста. После закрытия пакета — архивируем секцию в
  `staff/ledger/archive/PACKAGE-<id>.md` и обнуляем `CURRENT.md`.
- **Многозначность.** Один «Следующий PR». Если нужно два хода parallel — это
  отдельное решение, не норма. Ledger линеаризует.

## Деградация и safety

- **Codex видит конфликт между ledger и Slack.** Trust ledger. Slack для
  exception'ов, ledger — операционная норма. Если QA забыл обновить ledger
  после Slack-разговора, Codex ОДНОЙ строкой в Slack пингует «ledger не
  синхронизирован, перечитать?» и стоит.
- **Ledger говорит GO, но preflight'а нет.** Это нормально для первого PR
  пакета (Codex пишет preflight как первый шаг). Если для НЕ-первого PR пакета
  preflight отсутствует — Codex стоп и пингует в Slack: «PR-X в ledger как GO,
  но preflight не вижу, package-preflight тоже не покрывает — нужен раунд».
- **Ledger говорит GO на ветке, которая уже существует.** Это значит предыдущая
  попытка не закрылась. Codex `git fetch` ветку, читает что в ней, продолжает
  с того места. Не открывает PR заново.
- **Главное правило.** Ledger — не magic-pilot. Если что-то выглядит
  неправильно, лучше стоп + одна строка в Slack, чем «накатил по ledger'у и
  сломал».
