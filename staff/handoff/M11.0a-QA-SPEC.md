### H6. `WeaponInstance` тип не определён в §1

Вопрос 5 рекомендует multi-instance: каждый собранный ствол — отдельный объект с `instance_id`, `mods`, `durability`. Но **спека §1 не отражает это**:

```ts
interface Item {
  // ...
  attached_mods?: string[];  // ← это «runtime» поле на каталог-Item — концептуально неправильно
}
```

Должно быть **два типа:**
```ts
interface Item { /* каталог-определение, immutable */ }
interface WeaponInstance { /* runtime, в инвентаре, mutable */
  instance_id: string;
  item_id: string;       // ссылка на каталог
  mods: Record<ModSlot, string | null>;
  durability: number;
}
```

**Fix:** §1 переписать с двумя типами после закрытия вопроса 5 Alex'ом (B1).

### M6. Conflict between §1 default tier=1 и существующие items с tier=1..3

§1 говорит «`tier?: 1..5; дефолт: 1`». Но 80 существующих items.json УЖЕ имеют `tier: 1..3` явно проставленный.

При работе ItemRegistry: если item имеет `tier: 2`, должен ли он переопределить дефолт? Очевидно да (см. §11 `tier: raw.tier ?? 1`). Но это создаёт двусмысленность с новыми craft_*: если craft_shiv не имеет явного `tier`, он получит 1. А существующий `shiv` имеет `tier: 2`. Если оба останутся в каталоге — конфликт.

**Fix:** §1 уточнить: «новые items в каталоге М11.0+ ОБЯЗАНЫ иметь явный `tier`. Дефолт=1 только для legacy без tier (которых нет — все 80 имеют)».
