## Твои deliverables

### 1. `docs/GDD.md` §5 amendment — **5 новых типов мобов**

Дополни существующую §5 «Мобы» новой подсекцией **§5.4 «Мобы M3»**, описав 5 новых мобов. Каждый должен быть **механически уникален** (не клон существующих по поведению).

Правила:
- 2 моба для зоны **Склад** (warehouse).
- 2 моба для зоны **Город** (city).
- 1 моб «мост» (может встречаться в нескольких зонах — например, ещё один тип мутанта).
- Каждый моб должен иметь **уникальный AI-паттерн**, отличный от трёх существующих (marauder = убегает при HP<30%, wild_dog = атакует первым, mutant = тупо бьёт).

Примеры допустимых AI-паттернов (не обязательно ровно эти — придумай свои в рамках жанра):
- «Снайпер» (ranged-only, наносит больше урона на дистанции, слаб в melee).
- «Защитник» (берёт «Укрытие» каждый второй ход).
- «Берсерк» (HP < 50% → удваивает урон, но проигрывает в инициативе).
- «Стая» (всегда минимум 2 моба, координируют атаки — но это сложнее для Engineer).
- «Хилер» (при HP союзника < 30% использует свою «аптечку» — нужен mob.type «medic»).

ЗАПРЕЩЕНО:
- AI, требующий перков игрока (M4).
- AI, требующий радио-доверия (M6).
- Боссы / multi-stage / phase-changes (M5).
- Любая координация между мобами в пределах одной волны, если она требует нового runtime-state (это уже на грани сложности M3 — оставь на M5 ИЛИ опиши минимально).

Для каждого нового моба укажи:
- `id`, `name_ru`, `type` (human / animal / mutant / mech / etc — добавь нужные в JSON-схему §6.2 если нужно).
- HP / damage_min / damage_max / speed / xp / defense (если есть).
- Краткое словесное описание AI (3-5 предложений).
- Пример боевого сценария (что игрок видит на CombatScene).
- Ссылку на дроп-таблицу (но сами числа дропа — в `balance.md`).

### 2. `docs/GDD.md` §6.4 amendment — **2 новые зоны**

Дополни §6.4 «Zone» подсекцией **§6.4.M3 «Новые зоны M3»**, описав 2 новые зоны: **Склад** (warehouse) и **Город** (city).

Для каждой зоны:
- `id`, `name_ru`, `description` (2-4 предложения).
- 2-3 уровня глубины (`depths`), каждый со списком мобов и ресурсов.
- `unlock_condition` (как игрок открывает зону):
  - Forest — `"start"` (уже есть).
  - Warehouse — например, `"complete_forest_depth_2"` или `"player_level >= 3"` (но M3 не имеет уровней — выбери что-то, что Engineer может прочесть из GameState без новой системы; например, `"any_sortie_completed"` или хардкод «всегда открыто»).
  - City — что-то более сложное, например, `"any_warehouse_sortie_completed"`.
- Перечень ресурсов в каждой глубине — каждая зона должна иметь **минимум 1 zone-exclusive ресурс**, который нельзя добыть в других зонах (правило из main GDD «Зоноспецифичные ресурсы»).
  - Пример: Warehouse → `electronics`, `oil` exclusive. City → `medical_supplies`, `circuitry` exclusive.

### 3. `docs/GDD.md` новая §7 — **Radio system structure (UI-заглушка для M3)**

Создай новую секцию §7 «Радио и доверие — структура» (полная логика будет на M6, но структура нужна сейчас).

Что описать:
- `RadioSignal` JSON-схема (поля: `id`, `from`, `subject`, `body`, `options`, `expires_after_sorties`, etc).
- `RadioScene` UI-flow: вход (кнопка «Радио» на BaseScene) → список сигналов → выбор сигнала → 2 кнопки («Откликнуться» / «Игнорировать») → возврат на BaseScene.
- На M3: **обе кнопки только убирают сигнал из активных** (отметка `dismissed`/`accepted` в state), **БЕЗ реальных последствий**. Полная логика (засады, награды, репутация фракций) — M6.
- 2-3 dummy-сигнала как пример. Не пиши боевые сценарии, не пиши награды — только «текст письма + 2 кнопки».
- Явно укажи: «На M3 RadioScene = заглушка. Engineer реализует UI только. Содержимое сигналов = плейсхолдеры от Content. Полная логика — M6.»

### 4. `docs/balance.md` amendment — **числа для M3**

В отдельной секции «§M3 — числа» добавь:
- Stats новых 5 мобов (HP, damage, speed, xp, dropTable).
- Drop tables новых 5 мобов (item_id → chance, count_min/max).
- Numbers новых 10 рецептов (ingredients + result_count). 5 рецептов M2 НЕ трогай.
- Numbers новых items, которые понадобятся для рецептов (но weight items — на стороне Content; ты только укажи **что должно появиться**, числа weight выбирает Content согласно GDD §3 «Weight tiers»).
- `unlock_condition` triggers для зон — текстом, не формулой (Engineer переведёт в код).

### 5. Anti-scope reminder в начале amendment

В верхней части §5.4 / §6.4.M3 / §7 явно повтори, что **НЕ** входит в M3:
- Перки (M4).
- Боссы / инстансы / T3 чертежи (M5).
- Полная логика радио (M6).
- Модули оружия (M3 stretch — НЕ обязательно).

## Процедура

1. Клонируй репо, `git checkout m3-integration`, прочитай файлы из «Контекст».
2. Напиши **короткий план** (5-7 пунктов) → отправь Alex'у блокирующим: «План готов, жду апрува PM».
3. После апрува — `git checkout -b m3/gd-amendment`.
4. **Recovery-safe**: после первой 1-2 правки сделай первый push + открой Draft PR `m3/gd-amendment → m3-integration` (это спасёт работу при смерти сессии).
5. Дополни GDD §5.4 / §6.4.M3 / §7 + balance §M3.
6. Обнови `staff/status/GAME_DESIGNER.md` (что сделано, под M3).
7. Сообщи Alex'у блокирующим: «GD M3 amendment готов, PR <ссылка>, жду PM-ревью + QA Spec».

Token-budget: эта задача — ~30-60 минут чтения + 60-90 минут письма. Если приближаешься к 50% лимита — push partial + recovery block в PR, PM подхватит continuation.
