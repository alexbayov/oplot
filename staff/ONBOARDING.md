# Оплот — онбординг и план работ

> Для нового разработчика. Цель файла: за один проход понять, что мы строим,
> как устроен код, по какому процессу едут PR, где мы сейчас и что делаем дальше.
> Канон по вехам — `docs/ROADMAP.md`. Этот файл — точка входа, роадмап — глубина.

---

## 1. Что это за игра

**Оплот** — post-apoc shelter sim с глубоким крафтом и вылазками. Платформа —
Яндекс.Игры (HTML5, mobile-first, landscape 1280×720).

Северная звезда (зафиксирована пивотом M13, не пересматривается): игрок строит
убежище, вылазка — это сервис для базы, а не самоцель. Петля:

```
база хочет ресурс → вылазка тащит ресурс → база меняется → крафт усиливает вылазку
```

Три несущих столпа, на них висит всё M15+:
1. **Крафт** — компоненты → тиры → аффиксы → ремонт → salvage. То, ради чего возвращаются.
2. **База-сим** — постройки, офлайн-прогрессия, конвертация ресурсов как мотив для вылазок.
3. **Вылазка с выбором** — авторесолв + decision surface (цель, расходники, точка остановки, события).

Сравнения по жанру: Sheltered, Fallout Shelter, This War of Mine. **Не** Darkest
Dungeon, **не** Fallout-тактика.

**Anti-scope (вето, постоянно).** Тактический пошаговый бой (отдельный трек после
стабилизации ядра, не на критпути), коммуна/NPC и сюжетная кампания (пост-релиз),
PvP/мультиплеер, 3D/изометрия, фэнтези, реальные ТМ оружия. Если фича тянет сюда —
останавливаемся и обсуждаем, а не пишем код.

## 2. Стек и быстрый старт

Phaser 3 + TypeScript + Vite + Zod. Тесты на Vitest. Yandex Games SDK на платформенном слое.

```bash
npm install
npm run dev        # dev-сервер с HMR
npm run build      # tsc && vite build → dist/
npm run lint       # eslint src/
npm run typecheck  # tsc --noEmit
npm run test       # vitest run
```

**Четыре гейта, которые должны быть зелёными в каждом PR:**

| Гейт | Команда | Что ловит |
|---|---|---|
| types | `npm run typecheck` | `tsc 0` ошибок |
| lint | `npm run lint` | `eslint 0` на `src/` |
| tests | `npm run test` | весь vitest зелёный |
| build | `npm run build` | продакшн-сборка проходит |

Текущий baseline на `main` (`f02c2e5`): `tsc 0` · `eslint 0` · `vitest 592 passed` · `build ✓`.
Если у тебя локально не сходится с этим — не пиши код, сначала почини окружение.

> Node: тесты гоняем на Node 22. `build` даёт chunk-warning на Phaser-бандл (~1.6MB) —
> это pre-existing, не блокер, игнорируем.

## 3. Карта репозитория

```
oplot/
├── src/            # весь код (Phaser 3 + TS), ~133 .ts файла
│   ├── scenes/     # Phaser-сцены (экраны): база, вылазка, верстак, арсенал…
│   ├── systems/    # чистая игровая логика (крафт, durability, sortieResolve, офлайн…)
│   ├── state/      # сейв-модель, миграции SAVE_VERSION, persist
│   ├── types/      # доменные типы (item, zone, encounter, sortie, skillNode…)
│   ├── ui/         # переиспользуемые UI-компоненты + токены/лейаут
│   └── utils/      # лоадеры, мелочь
├── content/        # игровой контент в JSON (предметы, зоны, рецепты, события)
├── docs/           # ROADMAP, GDD, balance, style-guide, redesign/
├── staff/          # процесс, леджер, исторические handoff'ы
└── assets/         # графика, звук
```

**Граница, которую держим:** `systems/` — чистая логика без Phaser, её легко тестировать
юнитами. `scenes/` — Phaser и рендер. Новую механику пишем как чистую функцию в `systems/`
с тестами, сцена только дёргает её и рисует. Если логика залезла в сцену — её тяжело
покрыть, это сигнал вынести.

**Ключевые системы (где что лежит):**
- `systems/sortieResolve.ts` — авторесолв вылазки, тут считается исход боя/энкаунтера.
- `systems/craft.ts`, `weaponAssembly.ts`, `weaponAffixes.ts`, `armorAffixes.ts` — крафт и аффиксы.
- `systems/durability.ts`, `repair.ts` — износ и ремонт оружия.
- `systems/offlineProgression.ts` — аккруал «пока тебя не было», капы на серверном времени.
- `systems/loadout.ts` — выбор расходников на вылазку (вес делит вместимость с лутом).
- `systems/heroPassives.ts`, `skillEffectCoverage.ts` — дерево навыков и его эффекты в геймплее.
- `state/` — `SAVE_VERSION` и миграции `migrateVNToVN+1`. **Трогать осторожно** (см. §5).

## 4. Процесс: как едет PR

Мы работаем пакетами. Пакет = одна веха (или её кусок) = N PR, где N бьётся по швам
кода, а не по размеру. Одна вертикаль = один PR, ~300-800 LOC. «Вся веха одним PR» — нет.
«Микро-PR на три строки» — тоже нет.

Жизненный цикл одного PR:

```
preflight → ветка → код + тесты → 4 гейта зелёные → PR → ревью → merge --no-ff
```

1. **Preflight.** Перед кодом пишем `staff/implementation/<PR-id>-PREFLIGHT.md`:
   разбор по реальному коду (с указанием файлов и строк), какие design-форки решены и
   как, какой инвариант защищаем тестом. Code-grounded — не из воздуха. Это снимает
   споры в ревью: к моменту PR архитектура уже согласована.
2. **Ветка** от свежего `main`. Имя — `<автор>/<pr-id>-<слаг>`.
3. **Код + тесты.** Новое поведение = новый тест. Тест фиксирует инвариант, а не
   реализацию (сравниваем поведение, а не внутренние поля).
4. **Гейты.** Все четыре зелёные локально до открытия PR.
5. **PR** с описанием: что внутри, какие гейты, какой SAVE-эффект (bump/neutral).
6. **Ревью** на независимом клоне (ревьюер гоняет гейты у себя, не верит на слово).
7. **Merge `--no-ff`** — сохраняем импл-коммит в истории. После мержа ветка сносится.

### Леджер — чтобы не пинговать друг друга после каждого мержа

`staff/ledger/CURRENT.md` — единственный live-файл состояния пакета. В начале сессии
читаешь его: на каком `main` стоим, какой PR следующий, `Статус: GO` или `ОЖИДАНИЕ`,
какие форки пакета уже решены. Ревьюер обновляет `CURRENT.md` как часть мерж-ритуала
(помечает PR закрытым, переключает «следующий PR»). Так рутина «след. PR → preflight →
PR → ревью → merge» едет без раунд-трипа в Slack. Slack — только для исключений:
непредвиденный архитектурный форк, блокер, бизнес-call.

Контракт леджера целиком — `staff/ledger/README.md`. **`CURRENT.md` правит только ревьюер**,
автор кода его не редактирует.

## 5. Дисциплина миграций (самое опасное место)

Сейвы версионируются `SAVE_VERSION`. Любое изменение формы сейва = bump + миграция
`migrateVNToVN+1` + round-trip unit-тест (старый сейв → миграция → новый формат,
без потерь и падений).

Жёсткие правила:
- **Две вехи с bump'ом нельзя вести параллельно** — два пути `vN→vN+1` конфликтуют.
  Ведём строго последовательно (порядок — за бизнес-приоритетом Алекса), либо вторая
  ветка ребейзит миграцию поверх первой.
- **Unknown id на загрузке → no-op skip, не падать.** Если аффикс/узел убрали из реестра,
  старый сейв с ним должен грузиться.
- **Frozen value на инстансе всегда побеждает реестр.** Баланс-патч реестра не
  переписывает значения в уже сохранённых инстансах.

Если PR трогает `state/` или форму сейва — это всегда повод для отдельного внимания
на ревью.

## 8. Разделение обязанностей и взаимное ревью

Двое инженеров: **новый разработчик** (human) и **Ross** (Head of Automation). Базовый
принцип — каждая строка кода в `main` прошла через второго. Автор PR никогда не мержит
сам себя.

**Зоны (направление по умолчанию, не жёсткие стены):**

| Слой | Ведущий | Ревьюер |
|---|---|---|
| Геймплей-системы (`systems/`: крафт, вылазка, дерево, durability) | новый разработчик | Ross |
| Сейв-схема и миграции (`state/`, `SAVE_VERSION`) | Ross | новый разработчик |
| UI/сцены (`scenes/`, `ui/`) | новый разработчик | Ross |
| Процесс, леджер, preflight-дисциплина, гейты | Ross | новый разработчик |
| Контент (`content/*.json`), баланс-числа | по задаче | второй |

«Ведущий» пишет код и preflight, «ревьюер» гоняет гейты на независимом клоне и держит
инвариант. На любом PR роли назначаются в начале пакета и не меняются по ходу.

**Правила взаимного ревью:**
1. Ревью — на независимом клоне. Гейты гоняешь у себя, не веришь описанию PR на слово.
2. Ревью предметное: указывай файл и строку, проверяй инвариант, а не стиль.
3. `merge --no-ff` делает ревьюер после APPROVE. Импл-коммит сохраняется.
4. Миграции (`state/`) — всегда четыре глаза, кто бы ни вёл. Это правило сильнее таблицы выше.
5. Design-форк по продукту (не по архитектуре) → эскалация Алексу одним сообщением, не блокируем код.
6. Леджер `CURRENT.md` обновляет тот, кто мержит.

Разделение — стартовая рамка. После первого совместного пакета сверимся и подвинем
границы под то, что реально работает.

## 9. Первые шаги нового разработчика

1. Клонировать репо, `npm install`, прогнать все четыре гейта — убедиться, что baseline
   (`vitest 592`) сходится локально.
2. Прочитать `docs/ROADMAP.md` целиком и `docs/redesign/M13-PIVOT.md` (видение пивота).
3. Прочитать `staff/ledger/README.md` (контракт леджера) и `staff/ledger/CURRENT.md`
   (где пакет сейчас).
4. Прогреть `systems/sortieResolve.ts`, `systems/craft.ts`, `state/` — это сердце игры.
5. Взять первый узкий PR из M21-part-A (прямая проводка одного inert-узла дерева, например
   `extra_loot_1` → `rollLoot`) — маленькая вертикаль с понятным тестом, хороший вход.
