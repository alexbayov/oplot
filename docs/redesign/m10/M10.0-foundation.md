# M10.0 — Foundation

> Phase 0 для вехи M10 «Living Refuge». Без неё все остальные фазы стартуют наощупь.
> Версия: v1 · Автор: Zo · PM: Alex

---

## Зачем

В исходном M10-плане нет phase 0. Это создаёт три проблемы:

1. **Нечем измерить успех.** Цель M10 — «средняя сессия растёт». Но baseline-метрик нет: мы не знаем, сколько сейчас длится средняя сессия, сколько вылазок завершают, где отваливаются.
2. **Архитектура overlay-режима** для M10.3 (Craft/Inventory/Radio как hotspots в BaseScene) не выбрана. От этого зависит код всех subscene'ов. Если выбрать неправильно — придётся переписывать.
3. **Нет before/after** для PR-описаний. Нечего показать «было так — стало так».

M10.0 это закрывает за один PR.

## Что входит

### 1. Telemetry baseline (минимальная)

Не аналитическая платформа — **5 событий** в Yandex Metrica через Yandex SDK. Цель: получить 1 неделю baseline-данных до старта M10.1, чтобы было с чем сравнивать.

**События:**

| Событие | Когда | Параметры |
|---|---|---|
| `session_start` | Boot scene complete | `day_count`, `level`, `is_returning` |
| `sortie_started` | SortieScene → Combat | `zone_id`, `depth`, `weight`, `hp_pct` |
| `sortie_completed` | ReturnScene success | `zone_id`, `depth`, `loot_count`, `hp_remaining`, `duration_sec` |
| `combat_resolved` | CombatScene end | `outcome` (won/fled/died), `turns`, `damage_taken` |
| `session_end` | beforeunload / visibilitychange | `duration_sec`, `scenes_visited` |

**Реализация:** `src/systems/telemetry.ts` — простой wrapper над `ysdk.features.LoadingAPI?.ready()` + `ysdk.gameplayStart()` + custom event dispatch. Off-platform (локальный dev) → console.log.

**Не делаем:** funnel-анализ, retention cohorts, A/B, аттрибуция. Это потом, если данные покажут что нужно.

### 2. Scene stack architecture decision

**Решение:** **`scene.launch()` + `scene.pause()` (Phaser scene stack)**, **не** Container-overlays.

**Почему:**
- BaseScene остаётся живой (анимации, ambient SFX) когда поверх открыт Craft — это как раз feel «living refuge»
- Все 12 текущих сцен работают через `scene.start()` (полная замена) — добавление `launch()` рядом не ломает существующий код
- Дешевле миграция: subscene не переписывается, добавляется только entry point `launchAsOverlay()`
- z-depth решается через scene order, не через ручной depth-tracking в одном Phaser.Scene

**Альтернатива (отвергнута):** Container-based overlays внутри BaseScene. Преимущество — нет повторной инициализации сцены. Недостаток — BaseScene становится monolith на 2000+ строк, depth conflicts, потеря изоляции.

**Что строим в M10.0:**
- `src/systems/sceneStack.ts` — две функции:
  - `openOverlay(currentScene, overlayKey, data?)` — `scene.launch(overlayKey, data)` + `scene.pause(currentScene.key)`
  - `closeOverlay(overlayScene)` — `scene.resume(parentKey)` + `scene.stop(overlayScene.key)`
- Опциональный fade-tween (200ms) между переключениями
- Тест на 3 кейса: open, close, double-open (idempotent)

В M10.0 **не интегрируем** в существующие сцены — это работа M10.3. Просто кладём helper и тест.

## Что НЕ делаем в M10.0

- Не трогаем визуал. Никаких новых ассетов, никаких изменений в существующих сценах.
- Не делаем funnel-анализ в Metrica. Только raw events.
- Не интегрируем sceneStack в существующие сцены — это M10.3.
- Не пишем M10.1 (карта) код. Только подготовка.
