> **Статус:** черновик плана, ожидает подписи владельца.

> **Автор:** Ross (ross@bimross.com) совместно с Алексеем Баёвым.

## Зачем

Текущая игра позиционируется как «Survival RPG с пошаговым боем и глубоким крафтом». В реальности:

- бой собирался автоматически (ГПТ + Codex) без удерживающего видения, оброс комбат-модулями (`combat*` ≈ 30% TS-кода) и при этом остался плоским: 4 кнопки, один таргет, без статусов и без выбора;
- крафт декларирует «модульные компоненты → уникальные статы», а отдаёт 5 рецептов «N материалов → готовый предмет»;
- база сводится к 4 кнопкам-ярлыкам (`Мастерская / Инвентарь / Радио / Герой`) и не является самостоятельным геймплейным слоем;
- кор-петля = скрипт без выбора (одна зона открыта, фиксированная глубина, фиксированный лут), потолок прогрессии на 5-м уровне (~2 часа геймплея);
- позиционирование жанра задаёт игроку ожидания Fallout, которые MVP не вытягивает.

M13 переопределяет жанр и порядок работ под реальное видение автора:

1. **Бой — заглушка-авторесолв** на ближайшие месяцы. Тактика возвращается отдельным треком после того, как ядро игры устоится.
2. **Крафт — глубокий, Lineage-style.** Компоненты, тиры, аффиксы, очередь, ремонт. То, ради чего игрок возвращается.
3. **База — симуляторный слой.** Постройки, офлайн-прогрессия, ресурсы базы как мотив для вылазок.
4. **Жанр — post-apoc shelter с крафтом и вылазками.** Сравнения: Sheltered, This War of Mine, State of Decay (shelter loop), Fallout Shelter. НЕ Darkest Dungeon.

Игра становится симом строительства убежища, где вылазка — сервис для базы, а не самоцель.

## Формула авторесолва (PR-1)

Чистая функция, без сцен и без `GameState` мутаций. Принимает снимок героя и параметры вылазки, возвращает результат, который потом применяется одной операцией.

```
resolveSortie(input: {
  hero:    { hp, hp_max, level, weapon, armor, skill_combat },
  sortie:  { zone_id, depth, encounters_planned },
  rng:     SeededRng,
}) → {
  encounters: Array<{
    mob_id, narrative_lines: string[],
    hp_lost: number, loot_rolled: ItemStack[],
    injury?: { kind: 'arm' | 'leg' | 'head', duration_days: number },
  }>,
  totals: { hp_lost_total, loot_total, time_minutes_total },
  outcome: 'success' | 'retreat' | 'knocked_out',
}
```

Базовая логика (детали уточняются в PR-1):

- Урон по герою = `mob.danger * (1 - armor_reduction) * depth_multiplier * rng_jitter(0.85..1.15)`.
- Урон по мобу = `(weapon.damage_avg + skill_bonus) * rng_jitter`. Моб умирает за N ходов, считаем суммарно.
- Лут = `roll_loot_table(zone, depth, rng)` + бонус за лишние ходы (если герой не выбит).
- Травма = шанс при `hp_lost > hp_max * 0.4`, выбирается из таблицы, висит N игровых дней (применяется в следующих вылазках как `accuracy -X%` или подобное).
- Нарратив = 1-2 строки из `content/narrative.json` (новый файл) по тэгам `(zone, mob, outcome)`. Маленький каталог на старте: ~30-50 строк, легко расширять.

Тесты — на формулу: seeded RNG, фиксированный вход, точный ожидаемый выход. Без моков.

## Крафт через компоненты (PR-2 + PR-3)

Идея Lineage 2: материалы → компоненты → готовое оружие со случайным аффиксом. Между состояниями — очередь с таймером и шанс провала (с возвратом части материалов).

**Новая схема `items.json`** (PR-2, только данные + Zod-валидация, без UI):

```
items:
  materials:      { id, name, weight, stack_max, rarity }            # scrap, wood, electronics, gunpowder, cloth...
  components:     { id, name, slot, stats: {...}, sources: [...] }   # barrel, grip, stock, firing_pin, plate, strap...
  consumables:    { id, name, effect: {...}, weight }                 # bandage, medkit, food, water, fuel...
  finished:
    weapons:      { id, kind, slots: [...], base_stats, possible_affixes: [...] }
    armor:        { id, kind, slots: [...], base_stats, possible_affixes: [...] }
    tools:        { id, kind, use: {...} }                            # crowbar, lockpick, screwdriver
```

Слот = именованная позиция в готовом предмете (например, у оружия `barrel`, `action`, `stock`, `mod`). Компонент попадает в конкретный слот, его статы суммируются в готовое.

**Аффикс** = случайный модификатор, применяемый при крафте: `+15% accuracy`, `-10% reload`, `+1 кг weight`. 6-10 префиксов и 6-10 суффиксов на старте, тэгированные по типу оружия. При успешном крафте бросается 0-2 аффикса в зависимости от тира верстака.

**Очередь крафта** (PR-3): берём компоненты в работу → ждём N минут → готово. Уровень верстака влияет на время, шанс провала и количество аффиксов. Готовый предмет можно разобрать обратно в компоненты (с потерями). Сломанный предмет можно отремонтировать (PR-3 опц.).

**Что это даёт:** 8 компонентов × 8 слотов × 12 аффиксов = тысячи комбинаций из малого числа сущностей. Игрок «гриндит» не XP, а конкретный компонент или аффикс.

## База как симулятор (PR-4)

`BaseScene` перестаёт быть меню. Это **двор оплота с слотами под постройки**.

**MVP-постройки:**

| Постройка | Что делает | Вход | Выход | Время |
|---|---|---|---|---|
| Грядка | Производит еду | вода | 5 ед. еды | 30 мин |
| Верстак | Открывает крафт оружия/брони | энергия (от ген.) | — | пассивно |
| Генератор | Производит энергию | топливо | 10 ед. энергии | 1 ч |
| Койка | Восстанавливает HP вне вылазки | время | +5 HP / 10 мин | непрерывно |

**Офлайн-прогрессия:** игрок закрыл игру в 18:00, открыл в 22:00 → 4 часа прошло, грядка выдала урожай 8 раз, генератор накопил энергию, верстак закончил крафт, который стоял в очереди. На открытии — окно «пока тебя не было».

**Дерево построек:** в M13.x можно добавить 2-й тир (расширенная грядка, ремонтный стол, склад топлива), но это потом. Сейчас 4 базовых.

**Ресурсы базы** становятся **мотивом для вылазок**. Грядке нужна вода → в `zones.json` появляются loot table с канистрами воды. Генератору нужно топливо → в зоне `Склад` повышенная вероятность топлива. Без топлива генератор встанет, без энергии верстак не работает, без верстака игрок не крафтит. Это и есть петля.

## Карта и зоны (PR-5)

`MapScene` сейчас показывает 9 зон, открыта одна. На лаунч (после M13) открыты **3 зоны** с разными loot-профилями:

- **Лес** — еда, вода, дерево, тряпки. Низкая сложность.
- **Склад** — топливо, металл, электроника. Средняя.
- **Промзона** — компоненты, оружейные детали, редкие материалы. Высокая.

`zones.json` обогащается полем `loot_profile` (вероятности по категориям). `mobs.json` обогащается полем `zone_affinity` (где встречаются). Гейтинг по уровню остаётся, но открывается быстрее (зона 2 на 2-м уровне, зона 3 на 4-м).

Остальные 6 зон становятся **future content** в `archive/m14-zones.md`. Они не открываются и не светятся на карте до M14.

## Сцены — итоговый граф

```
BootScene → BaseScene (главный экран)
                ├→ Слоты построек (грядка, верстак, генератор, койка)
                ├→ InventoryScene (склад + экипировка)
                ├→ CraftScene (компоненты + очередь)
                ├→ RadioScene (нарратив, как раньше)
                └→ MapScene
                       └→ SortieScene (зона + глубина)
                              └→ EncounterScene (короткая интерлюдия, опц.)
                                     └→ resolveSortie() → ReturnScene (сводка)
                                            └→ LootScene (распределение в склад)
                                                   └→ BaseScene
```

`CombatScene`, `LevelUpScene`, `ProgressionScene`, `SkillTreeScene` — выпиливаются или переезжают в архив. Уровень и перки остаются как чистые системы (`xp.ts`, `perks.ts`), но интерфейс прогрессии встроим в `BaseScene` отдельной вкладкой позже.

## Что вне скоупа M13

Перечисляю явно, чтобы не было неоднозначностей:

- **Тактический бой.** Возвращается в M14 как отдельный трек (укрытия, точность, манёвры). Сейчас не трогаем.
- **Скилл-дерево и перки.** Системы остаются в коде, но UI прогрессии переезжает на потом. Сейчас 5 уровней с пассивными перками достаточно.
- **Графика и арт.** Идут отдельным треком (текущие painted ассеты + новые иконки для компонентов и построек). Не блокируют ни один PR M13.
- **Монетизация.** Текущие IAP (`starter_pack`, `gas_pack`, `disable_ads`) сохраняются. Сезонный пасс и косметика — после M13.
- **Дейлики/еженедельные челленджи.** После M13 — это становится проще, потому что база сама создаёт ритм.
- **Балансировка чисел.** Цифры из `balance.md` сохраняются как стартовые. Точная подстройка — после PR-5 и первого playtest.

## Риски

- **PR-1 ломает много импортов разом.** Поэтому он первый и большой по строкам (но большинство удаления). После него рост.
- **Crafting UI (PR-3) — самый сложный кусок UI в проекте.** Нужно продумать макет до начала кодинга. Сделаю отдельный mini-design в начале PR-3.
