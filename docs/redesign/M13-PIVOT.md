## Зачем

Текущая игра позиционируется как «Survival RPG с пошаговым боем и глубоким крафтом». В реальности:

- бой собирался автоматически (ГПТ + Codex) без удерживающего видения, оброс комбат-модулями (`combat*` ≈ 30% TS-кода) и при этом остался плоским: 4 кнопки, один таргет, без статусов и без выбора;
- крафт декларирует «модульные компоненты → уникальные статы», а отдаёт 5 рецептов «N материалов → готовый предмет»;
- база сводится к 4 кнопкам-ярлыкам (`Мастерская / Инвентарь / Радио / Герой`) и не является самостоятельным геймплейным слоем;
- кор-петля = скрипт без выбора (одна зона открыта, фиксированная глубина, фиксированный лут), потолок прогрессии на 5-м уровне (~2 часа геймплея);
- позиционирование жанра задаёт игроку ожидания Fallout, которые MVP не вытягивает.

M13 переопределяет жанр и порядок работ под реальное видение автора:

1. **Бой — заглушка-авторесолв** на ближайшие месяцы. Тактика возвращается отдельным треком после того, как ядро игры устоится.
2. **Крафт — глубокий, Lineage-style.** Компоненты, тиры, аффиксы, очередь, ремонт. То, ради чего игрок возвращается.
3. **База — симуляторный слой.** Постройки, офлайн-прогрессия, ресурсы базы как мотив для вылазок.
4. **Жанр — post-apoc shelter с крафтом и вылазками.** Сравнения: Sheltered, This War of Mine, State of Decay (shelter loop), Fallout Shelter. НЕ Darkest Dungeon.

Игра становится симом строительства убежища, где вылазка — сервис для базы, а не самоцель.

## Главный принцип последовательности PR-ов

Сначала валидируем **петлю** «база хочет ресурс → вылазка тащит ресурс → база меняется», и только потом углубляем крафт и достраиваем красивый UI базы. Иначе несколько недель проект живёт без проверенного нового core loop.

Конкретно: ресурсы базы (вода, топливо, металл, еда) появляются уже в PR-1 как счётчики с заглушками вместо построек, и зоны (PR-2) сразу приносят эти ресурсы. Полноценные постройки и офлайн-прогрессия — PR-5.

## Авторесолв с decision surface (PR-1)

Авторесолв ≠ «формула посчитала и показала текст». Это всё ещё скрипт. Игрок должен **управлять экспедицией**, иначе мы просто заменили скучный бой скучной таблицей.

**Точки выбора до запуска формулы:**

1. **Зона** (`MapScene`) — где лут и нагрузка отличаются.
2. **Глубина** (`SortieScene`) — больше глубина = больше лута и риска. Уже есть в коде.
3. **Цель вылазки** — короткий выбор перед стартом: `Тихо` (меньше лута, меньше травм), `Жадно` (больше лута, больше HP-loss), `Точечно: топливо` (бонус к конкретному ресурсу, меньше всего остального). Цель влияет на коэффициенты формулы.
4. **Экипировка** — оружие + броня уже выбраны в `InventoryScene`. Добавляем явный шаг «взять расходники» (бинты, аптечки, еду на дорогу). Что взял — то применяется автоматически в формуле, по чёткому правилу.

**Точки выбора во время вылазки:**

5. **Между энкаунтерами** — после каждой стычки короткий экран: `Идти дальше / Вернуться сейчас`. Возврат сейчас = меньше лута, но HP уже потерянное закрепляется. Игрок выбирает свою точку остановки.
6. **Случайное событие выбора** — 30% шанс на одну вылазку получить event с 2-3 опциями (`Найден тайник: открыть рискнув ловушкой / пройти мимо`). Это `content/narrative_events.json` с тэгами по зоне.

**Сигнатура формулы:**

```
resolveSortie(input: {
  hero:        { hp, hp_max, level, weapon, armor, skill_combat, injuries: [...] },
  sortie:      { zone_id, depth, goal: 'quiet'|'greedy'|'fuel'|'metal'|..., consumables: [...] },
  encounter:   { mob_id, narrative_tags: [...] },
  rng:         SeededRng,
}) → {
  narrative_lines: string[],   // 2-3 строки из narrative.json по тэгам
  hp_lost:         number,
  loot_rolled:     ItemStack[], // включая ресурсы базы
  consumables_used: ItemStack[],
  injury?:         { kind: 'arm'|'leg'|'head', duration_days: number },
  outcome:         'won' | 'fled' | 'knocked_out',
}
```

Функция вызывается **на каждый энкаунтер отдельно**, между ними — UI «Идти дальше / Вернуться». Сценарий вылазки = массив энкаунтеров + опц. event между ними + итоговая сводка в `ReturnScene`.

**Базовая логика (детали в PR-1):**

- Урон по герою = `mob.danger * (1 - armor_reduction) * depth_multiplier * goal_modifier * rng_jitter(0.85..1.15)`.
- Урон по мобу (для счёта победы/побега) = `(weapon.damage_avg + skill_bonus) * goal_modifier * rng_jitter`. Если мобу нужно > N ходов = шанс побега.
- Лут = `roll_loot_table(zone, depth, goal, rng)`. Цель `Точечно: топливо` сдвигает loot table в сторону топлива.
- Травма = шанс при `hp_lost > hp_max * 0.4`, из таблицы, висит N игровых дней (применяется в формуле через поле `hero.injuries`).
- Нарратив = 1-2 строки из `content/narrative.json` (новый файл) по тэгам `(zone, mob, outcome, goal)`.

**Тесты-инвариантности (gate для мержа PR-1):**

- Seeded RNG: фиксированный вход → фиксированный выход. 5-7 кейсов на разные комбинации `(zone, depth, goal, weapon)`.
- Цель `Точечно: топливо` действительно сдвигает loot в сторону топлива (статистический тест на 1000 ролов).
- Травма не выпадает если `hp_lost < hp_max * 0.4` ни в одном из 1000 ролов.
- Расходники применяются: герой со 100 HP + 1 аптечка теряет меньше HP чем без аптечки.

## Крафт через компоненты (PR-3 схема, PR-4 UI)

Идея Lineage 2: материалы → компоненты → готовое оружие со случайным аффиксом. Между состояниями — очередь с таймером и шанс провала с возвратом части материалов.

**Новая схема `items.json` (PR-3, только данные + Zod-валидация, без UI):**

```
items:
  materials:      { id, name, weight, stack_max, rarity }            # scrap, wood, electronics, gunpowder, cloth...
  components:     { id, name, slot, stats: {...}, sources: [...] }   # barrel, grip, stock, firing_pin, plate, strap...
  consumables:    { id, name, effect: {...}, weight }                # bandage, medkit, food, water, fuel...
  finished:
    weapons:      { id, kind, slots: [...], base_stats, possible_affixes: [...] }
    armor:        { id, kind, slots: [...], base_stats, possible_affixes: [...] }
    tools:        { id, kind, use: {...} }                            # crowbar, lockpick, screwdriver
```

Слот = именованная позиция в готовом предмете (у оружия: `barrel`, `action`, `stock`, `mod`). Компонент попадает в конкретный слот, его статы суммируются в готовое.

**Аффикс** = случайный модификатор, применяемый при крафте: `+15% accuracy`, `-10% reload`, `+1 кг weight`. 6-10 префиксов и 6-10 суффиксов на старте, тэгированные по типу оружия. При успешном крафте бросается 0-2 аффикса в зависимости от тира верстака.

**Acceptance criteria для крафта (gate для мержа PR-4):**

Глубина появляется не от комбинаторики, а от понятных trade-off-ов. PR-4 без этого не мержится:

1. **Preview итоговых статов.** До нажатия «крафт» игрок видит, что получит: итоговый damage/accuracy/weight по сумме компонентов. Без preview слепой рандом.
2. **2-3 осмысленные build-стратегии прописаны в `docs/redesign/M13-BUILDS.md`** до старта PR-4. Например: «лёгкий снайпер (приоритет accuracy и weight)», «тяжёлый штурм (приоритет damage)», «мусорщик (приоритет дешёвых компонентов)». Каждая стратегия — это набор компонентов и аффиксов, который игрок может собрать. Если игра не даёт построить хотя бы одну из них — крафт сломан.
3. **Salvage loop.** Любой готовый предмет разбирается обратно в компоненты с потерями (50-70% возврат). Это не «выкинуть», это «попробовать ещё раз». Без этого плохой аффикс = мёртвый груз.
4. **Recraft с сохранением слотов.** Если у игрока хороший ствол и плохое ложе, он может пересобрать оружие, поменяв только один компонент. Не «крафтить с нуля».
5. **Никакого «рандом ради рандома».** Шансы аффиксов и их статы прописываются в `content/affixes.json` явно. Минимум 30% шанс получить аффикс на тире 1 верстака, 60% на тире 2, 90% на тире 3.
6. **Очередь крафта** — берём компоненты в работу, ждём N минут, готово. Уровень верстака влияет на время, шанс провала и количество аффиксов.

**Тесты (gate PR-3):**
- Zod-схема валидирует `items.json` на загрузке. Битый JSON падает в `BootScene`, не в бою.
- Все 3 build-стратегии из `M13-BUILDS.md` собираются из существующих компонентов (тест запускает сборку, проверяет что предмет существует).

## Ресурсы базы и петля «база-вылазка» (PR-1 минимально, PR-5 полностью)

**В PR-1:** добавляем 4 счётчика ресурсов базы в `GameState`: `water`, `fuel`, `metal`, `food`. Отображаем их в `BaseScene` как простой панель (без иконок и анимации). Зоны (через `loot_table`) могут уже сейчас выдавать эти ресурсы — это валидирует петлю до того, как мы построим красивые грядки.

**В PR-2:** перепиcка `zones.json` под loot-профили, привязанные к ресурсам базы:
- **Лес** — `water` (родники), `food` (растения), `wood`.
- **Склад** — `fuel`, `metal`, `electronics`.
- **Промзона** — `components` (готовые компоненты для крафта), редкие материалы.

**В PR-5:** настоящие постройки на `BaseScene`. Каждая постройка превращает один тип ресурса в другой или восстанавливает герой:

| Постройка | Что делает | Вход | Выход | Время |
|---|---|---|---|---|
| Грядка | Производит еду | `water` | `food` × 5 | 30 мин |
| Верстак | Открывает крафт (PR-3/PR-4) | `fuel` (через генератор) | — | пассивно |
| Генератор | Производит энергию для верстака | `fuel` | `energy` × 10 | 1 ч |
| Койка | Восстанавливает HP вне вылазки | `food` × 1 / 10 мин | +5 HP / 10 мин | непрерывно |

**Дерево построек:** второй тир (расширенная грядка, ремонтный стол, склад топлива) — задел на M13.x, не в скоупе M13.

## Офлайн-прогрессия и капы (PR-5, спека сейчас)

Офлайн-прогрессия — самый частый источник эксплойтов (перевод часов, clock rollback) и багов сейвов. Прописываем правила сразу в плане, до кода.

**Источник времени:**

1. Первый выбор — серверное время через Яндекс.Игр SDK (`ysdk.serverTime`). Используем как «истину».
2. Fallback — локальное время с дельтой к серверному, кэшированной в `cloudSave`. Если `ysdk.serverTime` недоступен в открытии, считаем по дельте.
3. Если дельта неправдоподобна (отрицательная или > 30 дней) — отбрасываем как «нет данных», начисляем 0.

**Капы и инварианты:**

- **Max offline window:** 8 часов. Всё что больше — обрезается. Игрок ушёл на 3 дня и пришёл — получает оффлайн за 8 часов, не за 72.
- **Storage cap на каждое здание:** грядка хранит максимум 20 еды до того, как игрок придёт собрать; генератор — 30 энергии; койка — до `hp_max`. После cap производство останавливается. Игрок не может уйти на 8 часов и получить 480 единиц еды.
- **Clock rollback handling:** если `last_seen_at > now`, считаем дельту 0 и логируем `telemetry.offline_rollback`. Не наказываем игрока, но и не начисляем.
- **Минимум:** не начисляем меньше чем за 1 минуту (если игрок выходил-входил мгновенно — пропускаем).
- **Что пишется в cloudSave:** `last_seen_at` (серверное), `buildings: { id, accumulated_output, started_at }[]`, `hero: { hp, injuries }`.

**Тесты (gate PR-5):**
- Fake clock в тесте: ставим `last_seen_at = T-10h`, проверяем что начислили за 8h (cap сработал).
- Rollback: `last_seen_at = T+1h` → дельта 0, telemetry event.
- Storage cap: 100h оффлайн → грядка отдаёт ровно 20 еды.

## Миграция сейвов (PR-1)

Снос боя ломает игроков, которые сейчас в середине вылазки или в `CombatScene`. PR-1 без миграции не мержится.

**Контракт миграции:**

1. **Версия сейва** уже есть в `config.ts` (`SAVE_VERSION = 3`). Бампим до 4.
2. **Миграция v3 → v4** в `src/state/migrations.ts`:
   - Если игрок был в активной вылазке (`GameState.activeSortie != null`) — откатываем в базу с сохранением HP и инвентаря. Лут вылазки теряется.
   - Удаляем поля `currentMagazineByWeaponId`, `combatStatusesByTarget`, `distanceBand`, `currentNoise` — они больше не нужны.
   - Добавляем поля ресурсов базы со стартовыми значениями: `water=0, fuel=0, metal=0, food=0`.
   - Добавляем `last_seen_at` если отсутствует.
3. **Telemetry event:** `save.migrated_to_v4` с прежней версией и фактом был-ли-в-бою. Чтобы видеть масштаб эффекта на живых игроков.
4. **Тест миграции:** фикстуры в `src/state/__tests__/migrations.test.ts` — снапшот v3 в бою → миграция → снапшот v4 валидный + игрок в базе.

## Сцены — итоговый граф

```
BootScene → BaseScene (главный экран + ресурсы базы)
                ├→ [PR-5] Слоты построек (грядка, верстак, генератор, койка)
                ├→ InventoryScene (склад + экипировка + расходники)
                ├→ CraftScene (компоненты + очередь, переписан в PR-4)
                ├→ RadioScene (нарратив, как раньше)
                └→ MapScene
                       └→ SortieScene (зона + глубина + цель + расходники)
                              └→ EncounterScene (короткий заход, опц. event)
                                     └→ resolveSortie() → между энкаунтерами:
                                            «Идти дальше / Вернуться»
                                     └→ ReturnScene (сводка вылазки)
                                            └→ LootScene (распределение в склад/базу)
                                                   └→ BaseScene
```

`CombatScene`, `LevelUpScene`, `ProgressionScene`, `SkillTreeScene` — выпиливаются или переезжают в архив. Уровень и перки остаются как чистые системы (`xp.ts`, `perks.ts`), но интерфейс прогрессии встроим в `BaseScene` отдельной вкладкой позже.
