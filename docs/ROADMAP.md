## Северная звезда

**Жанр (зафиксирован пивотом M13): post-apoc shelter sim с глубоким крафтом и вылазками.**
Сравнения: Sheltered, Fallout Shelter, This War of Mine. НЕ Darkest Dungeon, НЕ Fallout-тактика.

Игрок строит убежище; вылазка — сервис для базы, а не самоцель. Петля:
**база хочет ресурс → вылазка тащит ресурс → база меняется → крафт усиливает вылазку.**

Три несущих столпа (всё M15+ висит на них):
1. **Крафт** — компоненты → тиры → аффиксы → ремонт → salvage. То, ради чего возвращаются.
2. **База-сим** — постройки, офлайн-прогрессия, конвертация ресурсов как мотив для вылазок.
3. **Вылазка с выбором** — авторесолв + decision surface (цель, расходники, точка остановки, события).

**Anti-scope (вето QA, постоянно):** тактический пошаговый бой (отдельный трек после стабилизации
ядра, не на критическом пути), коммуна/NPC и сюжетная кампания (пост-релиз, `staff/decisions/BACKLOG.md`),
PvP/мультиплеер, 3D/изометрия, фэнтези, реальные ТМ оружия.

## Схема карточки вехи

Каждая веха описана единым контрактом:
- **Цель** — одно предложение, зачем веха.
- **Фича на выходе** — что конкретно получает игрок.
- **PR-блоки** — 2-4 заголовка (заголовки, НЕ детализация). `веха = N PR, N по швам кода, не по размеру`:
  одна вертикаль = 1 PR; веха с 2-3 независимо тестируемыми вертикалями = N PR под общим preflight'ом;
  каждый PR 500-1200 LOC, не микро; «вся веха одним PR» — нет.
- **Система** — какой игровой слой растёт.
- **Схема** — влияние на `SAVE_VERSION`: bump (с миграцией `migrateVNToVN+1` + round-trip тест) или neutral.
- **Зависимости** — от каких вех (различать design-level и schema-level: две параллельные вехи с bump'ом конфликтуют миграциями).
- **DoD** — когда веха считается закрытой (acceptance, одно предложение).
- **Статус** — из легенды выше.

## 🔜 M15 — Жизненный цикл оружия и крафт-экономика

- **Цель:** у крафченого оружия полный управляемый цикл — поломка не тупик, а развилка чинить/разобрать.
- **Фича на выходе:** сломанное оружие чинится на верстаке за ресурсы/части; разбор перестаёт быть
  бесплатным (нельзя бесконечно craft↔salvage без потерь); в Арсенале видна дельта статов при смене оружия.
- **PR-блоки:**
  1. Repair core + UI (закрывает долг **C6** из `durability.ts`) — pure `repairInstance` + кнопка на верстаке/в Арсенале.
  2. Экономика разбора — частичный возврат частей / стоимость, чтобы craft↔disassemble имел трение.
  3. Арсенал: дельта-сравнение статов при equip-swap (UX).
- **Система:** durability/repair, крафт-экономика.
- **Схема:** neutral — repair мутирует те же 3 поля, что disassembly; refund-rate = логика; delta-compare = чистый UI. **NO SAVE bump.**
- **Зависимости:** M14.
- **DoD:** сломанное оружие можно починить на верстаке; разбор имеет стоимость/частичный возврат; в Арсенале видна дельта статов при equip-swap; vitest зелёный.
- **Статус:** 🔜 next.

## 📋 M16 — Глубина крафта: аффиксы, билд-стратегии, preview

- **Цель:** крафт даёт осмысленные trade-off-ы — три билд-стратегии из `M13-BUILDS.md` становятся собираемыми и читаемыми.
- **Фича на выходе:** части несут не только damage, а accuracy/weight (и аффиксы); перед сборкой —
  preview итоговых статов; видно, какую стратегию (Стрелок/Танк/Гриндер) собираешь.
- **PR-блоки:**
  1. Расширение stat-поверхности части (accuracy/weight сверх damage) + потребление их в `sortieResolve`.
  2. Preview-панель сборки — показать результат до commit.
  3. Аффиксы/качество-тир на компонентах (gate: тест собираемости 3 стратегий).
- **Система:** крафт-глубина, аффиксы.
- **Схема:** **v8→v9 (bump).** `ComponentItem.stats`/`WeaponInstance.stats` расширяются под accuracy/weight + affix; `migrateV8ToV9` с default-stamp на legacy-части + round-trip unit-тест.
- **Зависимости:** M15.
- **DoD:** Стрелок/Танк собираемы и различимы по статам через weapon-аффиксы; preview показывает результат до сборки; миграция round-trip-чистая. **Гриндер — после armor-affix пасса** (его идентичность — carry/scavenge на броне, не на оружии; в whitelist M16 не входит, см. preflight call G-A).
- **Статус:** 📋 planned.

## 📋 M17 — База как живой сим: постройки + офлайн-окна

- **Цель:** база — живой сим: постройки конвертируют ресурсы и копят офлайн, давая повод идти в вылазку.
- **Фича на выходе:** рабочий набор построек (генератор→energy, верстак-gating, койка→HP по часам);
  экран «пока тебя не было» при возврате; ресурс-сток, мотивирующий вылазки.
- **PR-блоки:**
  1. Производственный цикл построек (генератор/койка производят по часам).
  2. Офлайн-аккруал + summary-экран «пока тебя не было» (капы/anti-exploit на серверном времени).
  3. Баланс ресурс-стоков, чтобы вылазка была мотивирована.
- **Система:** база-симуляция, офлайн-прогрессия.
- **Схема:** likely neutral — `BuildingState.accumulated_output` и `accrueOffline` уже есть (M13-6c); возможен один timestamp-field на снапшот игрока, возможно нет.
- **Зависимости:** M15. **Design-level почти независима от M16, но schema-level — нет:** если обе вехи бампят `SAVE_VERSION`, два конкурирующих пути v8→v9 конфликтуют. Вести строго последовательно (порядок — за Алексом), либо вторая ветка ребейзит миграцию поверх первой.
- **DoD:** постройки производят ресурсы по часам; экран «пока тебя не было» показывает аккруал; ресурс-сток мотивирует вылазку; capы/anti-exploit на серверном времени.
- **Статус:** 📋 planned.

## Процесс

- **Drafter:** Виктор (dev) пишет v1, второй агент (QA) дополняет архитектурный слой (рост save-схемы,
  cloudSave-зависимости, баланс-TODO) и валидирует breadth-first срез против кода; Алекс — бизнес-приоритеты; мерж.
- **Ритм:** каждая закрытая веха = шаг + ревизия следующего горизонта (M18+ переписывается).
- **Дисциплина миграций:** две вехи с `SAVE_VERSION` bump'ом нельзя вести параллельно (конфликт путей `vN→vN+1`).
  Вести строго последовательно (порядок — за бизнес-приоритетом Алекса), либо вторая ветка ребейзит миграцию поверх первой.
- **Каждый PR:** preflight (`staff/implementation/<PR-id>-PREFLIGHT.md`) → gates (tsc/eslint/vitest/build) →
  QA на независимом клоне → merge `--no-ff`.
